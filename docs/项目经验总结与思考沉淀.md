# 环保危废仓库摄像头自动巡检系统 v3.0 — 项目经验总结与思考沉淀

> 覆盖周期：2026-07-06 ~ 2026-07-08（Phase 1 ~ Phase 5）

---

## 0. 核心结论

本项目将三套服务（环保小脑 / 鹊桥同步层 / 鹊桥 MCP 层）从架构骨架逐步推进至**基于真实海康凭证的实景验证**。相较代码交付本身，更值得沉淀的是以下两点关于"如何借助 AI 完成大型工程"的认知：

1. **流程约束并非对 AI 的限制，而是对主导者的保障。** 执行过程强制要求遵循 Superpowers 全流程（含 spec 审阅环节），即便 agent 认为过度繁琐。实践表明，正是由于要求 agent 先行落盘设计方案，当 Phase 4 其关于 MCP 的诸多假设被真实 SDK 推翻时，要"对照设计文档进行修正"，而非"在散落的代码中被动排查"。
2. **对"已完成"的表述须审慎分级。** agent 报告测试全绿、链路通过，一律视为"链路未中断"的初步信号。真正的验收，是在提供真实海康凭证、要求其完成实景抓拍并跑出真实数据之后方才确认。

## 1. 项目交付概览

本系统采用三层架构，负责危废仓库摄像头的自动巡检。

| 层级 | 服务 | 我职责 | 最终交付 |
|------|------|----------|----------|
| 环保小脑 | `enviro-brain`（8080） | 确立架构与同步契约 | 4 张数据表 + 9 个 API + 增量同步接口 + 定时巡检 + 飞书通知 + 台账 Word 生成 |
| 鹊桥·同步层 | `queqiao`（8081） | 确立"以实际代码契约为准"原则 | 4 张 synced 表 + 游标同步 + 单表容错 + notify 回调 |
| 鹊桥·MCP 层 | 同一应用内嵌 | 决策 SSE 传输 + 5 个工具 + 查询/操作分流 | 5 个 MCP 工具，脑机端仅经鹊桥取数、不穿透环保小脑 |
| 脑机端 | 外部桌面端 | 仅界定边界：仅允许经 MCP 访问 | —— |

**核心架构原则**为：脑机端仅与鹊桥 MCP 交互，禁止直连环保小脑。该原则在最终故障测试中（主动终止环保小脑进程、鹊桥服务仍维持 UP 状态）得到验证。

## 2. 方法论复盘：所坚持流程的效用评估

### 2.1 坚持 Superpowers 全流程的初衷

自项目启动，即确立准则：**任何代码变更，须先行经历 brainstorming → writing-plans → TDD → code-review → verify 全流程，即便细微改动亦须先出设计。** 此要求并非对 agent 能力的质疑，而是对其"假设的可靠性"存疑——agent 最具风险的情形，是以充分自信的姿态沿错误的预设持续推进。要求其先行落盘设计，等价于迫使其将隐含假设显性化，供我审阅。

### 2.2 流程实现托底的关键时刻

Phase 4 为典型例证。agent 初版设计陈述"`/mcp/enviro-inspection` + streamable-http"。未深究即准许实施——真实 SDK 反馈：webmvc starter 仅支持 SSE 传输，`transport: streamable-http` 直接映射为空路由（全 404），且 `base-url` 必须置空，否则客户端在 10 秒内无法获取 message endpoint。

**若未要求 agent 先行撰写 design，此类错误将深埋于代码之中，甚至无从判断其假设错在何处。** 正因设计文档在先，我的处置路径为：对照设计 → 要求其同步修订 design / plan / openspec / 代码，确保四处一致。设计文档在此充当了"agent 认知的版本控制"，得以基于版本差异施行审阅。

### 2.3 流程认知的修正

- **Superpowers 与 OpenSpec 并非二选一，而是前后衔接的两条 Pipeline。** Superpowers 属"输入阶段"——需求下达后执行 brainstorm，产出 `research.md`（调研依赖）与 `plans.md`（分步实施方案），且可在其中标注"本次变更涉及 N 个 Spec 变更"；编码完成后，OpenSpec 属"输出阶段"——将 `plans.md` 中已确定的决策（最终方案、API 签名、性能指标）脱水为精炼的 OpenSpec 条目，并在 `proposal.md` 末尾以 `> 决策依据详见：[link to superpowers/plans.md]` 回链。二者功能不等价、位置不同、顺序固定，而非冗余双写。
- **据此重新评估 Phase 4 的"补做 OpenSpec"：并非过度要求，而是补回被遗漏的输出环节。** 此前 agent 在 Superpowers 流程中产出了 `plans.md`，却未顺势进入 OpenSpec 输出阶段；我要求补做，实为补全流程，而非重复劳动。后续项目须将"代码跑通即执行 OpenSpec 更新 + proposal 回链"列为强制步骤。

## 3. 关键决策梳理

### Phase 1 — 同步契约为确立的地基
- **`sync_version_seq` 独立序列表 + `FOR UPDATE` 原子递增**，而非依赖自增主键。此为我的核心判断：增量同步的"水位"须保持单调、且与业务数据解耦。Phase 3 游标同步、Phase 5 回调均依赖此设计。
- 要求增量接口返回 `hasMore + nextSince`（游标分页），而非"获取 watermark 后全量拉取"。

### Phase 2 — 真实缺陷为要求冒烟所暴露
- 我要求执行**端到端冒烟（真实 MySQL + 真实飞书 webhook）**，由此暴露两项单元测试无法覆盖的缺陷：
  - `findActive(0,10000)` → MySQL `LIMIT -10000` 语法错误（page=0 致负 offset）。
  - 飞书通知手工拼接 JSON，traceback 与反斜杠及真实换行污染报文 → 飞书返回 400。
- **坚持"真实冒烟"于此得到验证。** 上述缺陷若仅依赖单测，将长期隐匿。据此确立准则：手工拼接 JSON 属反模式，一律要求采用 Jackson 序列化。

### Phase 3 — 确立"以实际代码契约为准"
- 方案文档 v3.0 已过时（陈述 `watermark` 字段、`record_id` 列；真实代码为 `SyncResponse{hasMore,nextSince}`、`record_id`）。**以运行的代码为权威，不以过时方案文档为准。**
- 要求同步层必须具备"游标退化防护 + 单表容错隔离"——因同步系长周期任务，单表失败不应波及全局。此逻辑后成为故障测试得以成立的基础。
- Maven 于 Git-Bash 下频现异常，我要求将其沉淀为 `maven-windows-build` skill——**工具链问题一旦解决，即须固化为可复用资产，避免重复踩坑。**

### Phase 4 — 决策架构，agent 验证，把关验收
- 决策：嵌入既有 queqiao 应用、采用 Spring AI MCP、**SSE 传输**、**5 个工具**、查询类直读 synced 库（不穿透）、操作类转发并友好降级。
- agent 遭遇三项 MCP 重大偏差（`MethodToolCallbackProvider` 静默丢弃返回 Object/Map 的 `@Tool` 方法、`base-url` 必空、webmvc 仅支持 SSE）。
- **据此确立管理原则：对其"单元 / Mock 通过"的结论一律存疑，仅认可"线缆上真实跑通"的验证结果。**

### Phase 5 — 验收逐关把控
- 首轮运行 agent 报告"6/6 PASS"，通过审阅报告，甄别出 2 项 P1（trigger 同步阻塞约 4 分钟、回调缺失 X-API-Key → 401）与 3 项 P2。要求建立修复链，复测通过后再行确认。
- **warmup 参数断点藏于中间层**（脚本已支持但未于 CLI 暴露、Service 未透传），单测无法发现。要求端到端贯通，并恢复对应断言。
- **提供真实海康凭证**（host / port / app-key / app-secret），要求其完成实景运行。沙箱缺凭证时，agent 仅能证明"代码路径通畅"，无法证明"真实抓拍可用"；注入真实凭证后跑出 115 路在线，确认"100% 完成"。

## 4. 技术陷阱速查

| 陷阱 | 现象 | 正确处置 |
|------|------|----------|
| MCP webmvc 传输 | `transport: streamable-http` → 全 404 | 采用 `WEBMVC`（SSE）；`GET /mcp/sse` + `POST /mcp/message` |
| `base-url` | 非空 → 客户端 10s 内未获取 message endpoint | `base-url: ""` |
| `@Tool` 返回类型 | 返回 Object/Map 被静默丢弃 | 必须返回具体 DTO（如 `OperationResultView`） |
| SSE 集成测试构造 | 默认 `sseEndpoint="/sse"` 易误 | `HttpClientSseClientTransport.builder(baseUri).sseEndpoint("/mcp/sse").build()` |
| H2 vs MySQL 方言 | `@MybatisTest` 默认 H2，`ON DUPLICATE KEY` 失效 | `@AutoConfigureTestDatabase(replace = NONE)` 启用 MODE=MYSQL |
| REST 客户端 Jackson | 默认 `FAIL_ON_UNKNOWN_PROPERTIES=true` → 反序列化失败 | 新建 ObjectMapper 禁用该特性 |
| 分页 indexed 契约 | page=0 → 负 offset 语法错误 | 接口明确 1-indexed，单测覆盖边界 |
| 手工拼接 JSON | 换行/反斜杠污染 → 400 | 一律采用 Jackson 序列化 |
| Git-Bash 终止 JVM | `kill` 报 No such process 且无法终止 | `taskkill /PID <winpid> /F /T` |
| GitHub SSH(22) 被防火墙拒绝 | `Connection refused` | 一次性 HTTPS URL 推送，保留原 SSH remote |
| Maven Windows 工具链 | `ClassNotFoundException` / cmd 不执行 | PowerShell + `mvn.cmd --%` + 项目级 `ci-settings.xml` |

> 上述陷阱已要求记入 `MEMORY.md`，后续同类项目直接复用，避免重蹈。

## 5. 对 AI agent 的协作管理

**作为人类，如何借助 AI agent 将大型项目正确交付**。

1. **我对其作出的两次纠偏，均已固化为准则**：
   - "不得跳过 spec 审阅"——其曾于我示意"直接执行"时跳过设计审阅径直动手，被我叫停。**我所定规矩：即使用户示意直接实施，亦须先经我审阅 spec。** 已写入项目约定。
   - "Phase 4 为何未做 OpenSpec"——其误将 OpenSpec 视为与 Superpowers 并列的"另一套流程"，我纠正：OpenSpec 是 Superpowers 之后的**输出阶段**（代码跑通后脱水 `plans.md` 决策 + `proposal.md` 回链）。要求补做，实为补回被遗漏的环节，并非冗余。**所得：须将"proposal.md 末尾回链 superpowers/plans.md"列为硬性规范，使两份制品形成可追溯的因果链。**

2. **代码推送由我自行处理。** 沙箱无 GitHub 凭证，我要求其本地提交、于自有终端推送。此系分工默契，亦为安全边界——其不接触我远端凭据。

3. **对"已完成"实施三级验收，不笼统采信**：
   - 代码完成 → 其报告"写完"仅作信息参考；
   - 链路通过 → 其跑通 Mock / 单测，我存疑；
   - 真实凭证验证通过 → 我方予确认。
   此分级意识，系 Phase 5 由"首轮 PASS"推进至"实景 113 在线"所淬炼而成。

4. **授予其探索权，同时划定红线**：细微修补由其自主决定；凡涉架构、对外契约、生产配置者，须由我决策。其提方案、我决策，而非由其代行决策。

## 6. 若重历本项目，我的改进方向

- **针对 MCP 这类"agent 不熟、且文档不可信"的技术，我将于设计阶段即勒令其先行 Spike**（线缆级探明 SSE-only / base-url / DTO 返回），而非待其撰写大量代码后再行返工。Phase 4 至少两轮返工本可规避。
- **将"真实外部依赖冒烟"列为各阶段强制验收 gate**，而非"有余力再行"。本次系被动发现缺陷，后续应由我主动设计冒烟。
- **严格遵循"Superpowers 输入 → OpenSpec 输出"的衔接**：代码跑通后立即执行 OpenSpec 更新，将 `plans.md` 决策脱水为条目，并在 `proposal.md` 回链 `plans.md`。此前我误判二者为二选一、要求"择一为主"，实为认知错误——二者本就是前后两段，缺一不可。
- **细微改动免除完整 design**：以简短 plan 加 commit 说明即可，不耗其设计仪式。
- **同步契约自 Day 1 即作 backbone**：此点我本已落实（`sync_version_seq` 独立表），后续须更早将其作为跨阶段共识钉死。
